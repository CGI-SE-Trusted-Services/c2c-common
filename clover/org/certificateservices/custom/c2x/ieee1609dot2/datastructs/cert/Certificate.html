28    * This structure is used for both implicit and explicit certificates.
29    *
30    * The fields in this structure have the following meaning:
31    * <p>
32    * <li>version contains the version of the certificate format. In this version of the data structures, this field is set to 3.
33    * <li>type states whether the certificate is implicit or explicit. This field is set to explicit for explicit certificates and to implicit for
34    * implicit certificates. See ExplicitCertificate and ImplicitCertificate for more details.
35    * <li>issuer identifies the issuer of the certificate.
36    * <li>toBeSigned is the certificate contents. This field is an input to the hash when generating or verifying signatures for an explicit certificate,
37    * or generating or verifying the public key from the reconstruction value for an implicit certificate. The details of how this field are encoded are
38    * given in the description of the ToBeSignedCertificate type.
39    * <li>signature is included in an ExplicitCertificate. It is the signature, calculated by the signer identified in the issuer field, over the hash
40    * of toBeSigned. The hash is calculated as specified in 5.3.1, where
41    * <br>
42    * -Data input is the encoding of toBeSigned following the COER.
43    * <br>
44    * -Signer identifier input depends on the verification type, which in turn depends on the choice indicated by issuer. If the choice indicated by issuer
45    * is self, the verification type is self-signed and the signer identifier input is the empty string. If the choice indicated by issuer is not self, the
46    * verification type is certificate and the signer identifier input is the canonicalized COER encoding of the certificate indicated by issuer. The
47    * canonicalization is carried out as specified in the ENCODING CONSIDERATIONS section of this subclause.
48    * <p>
49    * <b>ENCODING CONSIDERATIONS:</b>When a certificate is encoded for hashing, for example to generate its
50    * HashedId8, or when it is to be used as the signer identifier information for verification, it is canonicalized as follows:
51    * <li>The encoding of toBeSigned uses the compressed form for all elliptic curve points: that is, those points (which in this standard are all EccP256CurvePoints)
52    * indicate a choice of compressed- y-0 or compressed-y-1.
53    * <li>The encoding of the signature, if present and if an ECDSA signature, takes the r value to be an EccP256CurvePoint indicating the choice x-only.
54    *
55    * @author Philip Vendil, p.vendil@cgi.com
