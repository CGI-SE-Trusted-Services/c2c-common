27    /**
28    *This structure defines how to encode a generic secured message:
29    *
30    * <li> protocol_version specifies the applied protocol version. For compliance with the present document,
31    * protocol version 1 shall be used.
32    * <li> security_profile specifies the security profile for this secured message. The profiles define the contents
33    * of the variable header, payload and trailer fields. A message that does not conform to the profile is invalid. The
34    * default value shall be set to 0, if no specific profile is used.
35    * <li> header_fields is a variable-length vector that contains multiple information fields of interest to the
36    * security layer. If not defined otherwise in a message profile, the sequence of header fields shall be encoded in
37    * ascending numerical order of their type value.
38    * <li> payload_fields is a variable-length vector containing the message's payload. Multiple payload types in
39    * one message are allowed.
40    * <li> trailer_fields is a variable-length vector containing information after the payload, for example,
41    * necessary to verify the message's authenticity and integrity. If not defined otherwise in a message profile, the
42    * sequence of trailer fields shall be encoded in ascending numerical order of the type value.
43    * <p>
44    * Further information about how to fill these variable-length vectors is given via security profiles in clause 7.
45    *
46    * @author Philip Vendil, p.vendil@cgi.com
